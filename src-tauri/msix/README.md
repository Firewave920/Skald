# Skald MSIX packaging (Microsoft Store — "Path B")

Builds a Microsoft-Store `.msix` so **the Store signs the package** and Skald
avoids buying/renewing a code-signing certificate. This is the "Path B" route
from the *Skald Microsoft Store Publishing Guide*; the *MSIX Packaging
(Microsoft Store) Roadmap* explains why this hand-packed approach was chosen over
Tauri's (nonexistent) MSIX target, `cargo-packager` (no MSIX support), and the
MSIX Packaging Tool (VM installer-capture — too fragile for CI).

It is **Store-only** and deliberately separate from `pnpm tauri build`, which
keeps producing the signed NSIS installer for GitHub Releases ("Path A").

## Files

| File | Purpose |
|---|---|
| `AppxManifest.xml` | Package manifest **template**. `build-msix.ps1` substitutes the `__TOKEN__` values at pack time — don't hand-edit the tokens. |
| `build-msix.ps1` | Stages the runtime layout + assets + manifest and runs `makeappx pack`. |
| `github-workflow-msix.yml` | Optional CI workflow template (Phase 4). Drop into `.github/workflows/` to enable. |

## Prerequisites

- A **release build** exists at `src-tauri/target/release/` (run `cargo build --release` or `pnpm tauri build` first). The script copies the exe + bundled LibVLC/plugins + `ffprobe`/`tone` from there — the same layout the app resolves at runtime via `resource_dir()`.
- The **Windows 10/11 SDK** (`makeappx.exe`, `signtool.exe`). It ships with the VS Build Tools "Desktop development with C++" workload that Tauri already requires. The script auto-locates the newest SDK under `Windows Kits\10\bin`.
- Store tile assets in `src-tauri/icons/` (`Square150x150Logo.png`, `Square44x44Logo.png`, `Square71x71Logo.png`, `StoreLogo.png`) — already generated by `tauri icon`.

## Usage

```powershell
# From src-tauri/msix/ (or anywhere — paths resolve relative to the script):

# 1) Unsigned package for Partner Center submission (Microsoft signs at ingestion).
#    Pass the identity values Partner Center assigns once the app name is reserved:
./build-msix.ps1 -IdentityName "1234Publisher.Skald" -Publisher "CN=ABCD1234-..." -PublisherDisplayName "Your Name"

# 2) Locally-installable test package (for Add-AppxPackage on a clean machine/VM).
#    Add-AppxPackage refuses an unsigned package, so sign with a self-signed cert
#    whose subject EXACTLY equals -Publisher (see "Local install testing" below):
./build-msix.ps1 -Sign -CertPath .\skald-test.pfx -Publisher "CN=SkaldTest"
```

Output: `src-tauri/target/Skald_<version>_<arch>.msix` (gitignored).

The four-part version is derived from `tauri.conf.json`'s `version` (`1.0.0` ->
`1.0.0.0`); override with `-Version`. **Each Store update must strictly increase
this** — the version bump rides on the normal `tauri.conf.json` bump, so it can't
silently stall.

## Partner Center identity (placeholders)

`AppxManifest.xml` ships with **placeholder** identity values so the script runs
(and packs/installs locally) before a Partner Center account exists. Partner
Center **rejects** a package whose identity doesn't match the reserved app, so
before the first real submission:

1. Create the Partner Center account and **reserve the app name** ("Skald").
2. From the app's *Product identity* page, copy the assigned **Package/Identity
   Name**, **Publisher** (`CN=...`), and **Publisher display name**.
3. Pass them as `-IdentityName` / `-Publisher` / `-PublisherDisplayName`.

The script warns whenever a placeholder identity is still in use.

## Local install testing (self-signed)

`Add-AppxPackage` requires a signed package and a *trusted* signing cert. For
testing only (never ship this signed artifact — Partner Center re-signs):

```powershell
# Subject MUST equal the -Publisher you pass to build-msix.ps1.
$cert = New-SelfSignedCertificate -Type Custom -Subject "CN=SkaldTest" `
  -KeyUsage DigitalSignature -FriendlyName "Skald MSIX test" `
  -CertStoreLocation "Cert:\CurrentUser\My" `
  -TextExtension @("2.5.29.37={text}1.3.6.1.5.5.7.3.3", "2.5.29.19={text}")
# Export to PFX:
$pwd = ConvertTo-SecureString -String "test" -Force -AsPlainText
Export-PfxCertificate -Cert "Cert:\CurrentUser\My\$($cert.Thumbprint)" -FilePath .\skald-test.pfx -Password $pwd
# Trust it (admin): import the cert into Local Machine Trusted People.
# Build + sign + install:
./build-msix.ps1 -Sign -CertPath .\skald-test.pfx -CertPassword test -Publisher "CN=SkaldTest"
Add-AppxPackage .\..\target\Skald_1.0.0.0_x64.msix
```

Then verify under package identity: app launches; **Credential Manager** token
read/write still works; offline downloads + local-library scanning work; LibVLC
plugins load from the packaged location; first-run is OK with no WebView2
preinstalled (WebView2 Evergreen Runtime is a machine prerequisite — preinstalled
on Win11).

## WACK (Windows App Certification Kit)

Run WACK locally against the `.msix` before submitting. The publishing guide
flags a known **S-Mode "Package Sanity Test" blocked-executable** failure on
Tauri v2 apps. Skald bundles three helper executables that are the likely
candidates: `vlc-cache-gen.exe`, `ffprobe.exe`, `tone.exe`. If WACK flags them:

- `ffprobe.exe` / `tone.exe` are invoked at runtime (local-library metadata
  read/write) — they must stay; document their use if certification queries them.
- `vlc-cache-gen.exe` is only used once to build `plugins.dat`. You can
  **pre-generate `plugins.dat`** into `vlc-dist/plugins/` and drop
  `vlc-cache-gen.exe` from the payload (remove it from `$RuntimeFiles` in the
  script) so it isn't in the package at all. The runtime already skips cache-gen
  when `plugins.dat` exists (`lib.rs` setup).

## CI (Phase 4)

`github-workflow-msix.yml` is a template, not active (this repo has no CI yet).
To enable: copy it to `.github/workflows/`, set the identity secrets, and it will
produce the `.msix` as a build artifact on tagged releases. Partner Center
submission itself stays manual.
